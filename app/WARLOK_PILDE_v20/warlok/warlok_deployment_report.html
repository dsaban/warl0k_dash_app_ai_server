01 Deployment Topology

// Physical and logical placement of all WARLOK components on client site

Network Zones

WARLOK is deployed across three isolated network zones. The hub occupies the secure operations zone. Peers sit in the execution zone — the segment closest to the work being performed. The client console (browser-based UI) is in the staff zone, separated by a firewall that permits only HTTPS to the hub and read-only API calls.

YubiHSM2 Provisioning

Provisioning is a one-time operation performed offline, before connecting hub-node to the network. The provisioner engineer must sign the provisioning log. The default authkey password is changed immediately. The provisioning session is not repeatable — if the device is reset, all signed blocks from prior epochs become unverifiable without the backup epoch key record.
1
Install YubiHSM2 connector daemon
apt install yubihsm-connector && systemctl enable --now yubihsm-connector
2
Generate root HMAC key in slot 0x0010
yubihsm-shell: generate hmac-key 0 0x0010 warlok-root 1 sign-hmac,verify-hmac hmac-sha256
3
Create hub authkey (slot 0x0002) — sign-hmac only
put authkey 0 0x0002 warlok-hub 1 sign-hmac <generated-password-32chars>
4
Create per-peer authkeys (slots 0x0003–0x0005) — verify-hmac only
put authkey 0 0x0003 peer-alpha 1 verify-hmac <peer-alpha-password>
⚠
Change default admin authkey password immediately
change-authentication-key 0x0001 <new-admin-password-64chars> — store offline only
6
Record provisioning log — sign with admin authkey
Export: get-object-info for all provisioned keys; sign log file; store in offline vault
✓
Verify provisioning — run smoke test
python3 hsm/provisioner.py --verify — confirms sign/verify round-trip on device

03 Peer Node Setup

// Deployment of exec-node-1, exec-node-2, exec-node-3

Each peer node runs an independent WARLOK peer process. The installation is identical across all three; only the peer ID, port, and authkey password differ. Peers do not need a YubiHSM2 device — they authenticate to the hub's device using their designated authkey over the existing yubihsm-connector session on hub-node.

In the pilot configuration, peer HSM operations are proxied through the hub's connector. This means exec-node-1..3 send their sign/verify operations to hub-node:12345 via the authenticated session. For a hardened multi-site deployment, each peer would carry its own YubiHSM2 USB device. The adapter code supports both modes via the WARLOK_HSM_CONNECTOR env var.

Transport & TLS

Self-signed certificates are generated for the pilot. Each node gets a cert with its hostname as the Common Name. All nodes pin each other's certificates — no public CA is involved, eliminating the dependency on external PKI during a controlled site pilot.

TLS certificate generation — run on each nodeShell
# Generate cert + key for hub-node
openssl req -x509 -newkey rsa:4096 -keyout hub.key -out hub.crt \
  -days 365 -nodes -subj "/CN=hub-node" \
  -addext "subjectAltName=IP:<HUB_IP>,DNS:hub-node"

# Distribute hub.crt to all peer nodes as trusted CA
# Distribute peer-*.crt to hub-node for mTLS verification

# Verify cert fingerprint is logged in provisioning record
openssl x509 -in hub.crt -noout -fingerprint -sha256

Storage Targeting

Each peer writes execution results, replicated blocks, and telemetry to its own isolated filesystem path under /data/warlok/<peer-id>/. The FileStore backend uses SHA-256-derived filenames, making directory enumeration uninformative without the key index. Separate filesystem mount points per peer are recommended to bound any runaway write.

File ownership must be warlok:warlok with mode 750 on all store directories. The systemd service runs as the warlok user with ProtectSystem=strict and only the store path whitelisted via ReadWritePaths. Any attempt by the peer process to write outside its designated store directory will be blocked at the kernel level.

04 Communication Lines & Choke Points

// Two distinct lines of communication — each with defined choke points and test triggers

WARLOK has two functionally separate communication lines. Line A is the execution path: a signed command travels from hub to a peer, executes, produces a block, and the block hash returns with a storage receipt. Line B is the data transit path: raw bytes (telemetry, sensor data, results) move from a source peer to a designated storage peer, are checksummed, and a receipt is issued. Both lines are audited by the event log. Both lines terminate at the filesystem. Both are blocked at separate firewall chokepoints.

Line A — Execution Command Path

1
Hub: task constructed & HMAC-signed (transport layer)
Message envelope created: msg_id=UUID[:12], sender_id="hub", HMAC(root_key, canonical_json)
2
CHOKE POINT A-1 — Hub firewall egress to peer port
Hub → exec-node-X:800N via HTTPS/TLS. If this port is blocked, task never arrives.
3
Peer: TLS handshake + message received
mTLS: peer verifies hub cert fingerprint. Hub verifies peer cert fingerprint. Any cert mismatch → connection dropped.
4
Peer: transport HMAC verified
PeerNode._verify_msg() — HMAC(sender.root_key, msg.canonical()) vs msg.signature. AUTH_FAIL if mismatch.
5
Peer: task_runner executed
Result bytes produced. On exception: result = error message bytes, exec_status = ERROR.
6
CHOKE POINT A-2 — Peer HSM session (YubiHSM2 via connector)
Peer requests block signing from YubiHSM2. HKDF epoch→block key derived on-device. If connector unreachable or authkey invalid → signing fails, block not created.
7
Peer: block created + signed + inserted into peer DAG
Block hash = SHA-256(parents + payload + meta + timestamp). Signature = HMAC(block_key, hash_bytes). epoch stamped on meta._epoch.
8
CHOKE POINT A-3 — Peer store write (filesystem)
FileStore.put("task:<msg_id>", result_bytes). SHA-256 checksum computed. Written to /data/warlok/<peer>/<hash>.json. If filesystem full or permission denied → STORE_PUT fails, receipt not issued.
✓
ACK returned to hub: block_hash + store_key + checksum
Signed ACK message traverses CHOKE A-1 in reverse. Hub receives execution proof.

Line B — Data Transit Path

1
Source peer: TELEMETRY message assembled
data encoded as hex bytes. key = "sensor:<id>". Message HMAC-signed by sender.
2
CHOKE POINT B-1 — Source → target peer firewall
Direct peer-to-peer HTTPS or hub-mediated POST /peers/telemetry. Firewall must permit peer port in both directions.
3
Target peer: message received, transport HMAC verified
Same AUTH check as Line A. Sender must be a registered peer (correct root key).
4
CHOKE POINT B-2 — Target store write (filesystem)
FileStore.put(key, data_bytes). SHA-256 checksum assigned. Written to /data/warlok/<target>/. Filesystem quota, permissions, and disk space are all failure modes here.
5
StorageReceipt issued: key + checksum + backend + peer_id + timestamp
Receipt propagates back through CHOKE B-1 to the original sender as ACK payload.
6
CHOKE POINT B-3 — Receipt integrity verification
Sender stores receipt. At any future point: GET /storage/<peer>/get/<key> → StorageRecord.verify() re-hashes value vs stored checksum. STORE_CORRUPT event if mismatch.

Choke Point Inventory

Choke Point A-1
Hub → Peer Network Boundary
Firewall rule between secure ops zone and execution zone. Controls whether tasks can be dispatched at all. Bidirectional — also gates ACK responses.
TEST: Block port 8001 → dispatch task → confirm timeout + PEER_SEND_ERR event. Unblock → retest → confirm task executes.
Choke Point A-2
YubiHSM2 Connector Availability
Peer HSM operations route through the connector at localhost:12345 on hub-node. If yubihsm-connector is stopped, every signing operation fails — no block can be created by any peer.
TEST: systemctl stop yubihsm-connector → dispatch task → confirm "HSM unavailable" error in event log. Restart → confirm recovery.
Choke Point A-3 / B-2
Peer Filesystem Write Path
Each peer's FileStore writes to /data/warlok/<peer>/. Filesystem full, incorrect ownership, or SELinux/AppArmor denial will produce STORE_PUT failure. The operation fails cleanly — no partial writes — but the block is still created in the peer DAG without a storage receipt.
TEST: chmod 000 /data/warlok/beta → dispatch task to peer-beta → confirm STORE_PUT fails in events, block_hash still returned. Restore permissions → confirm next task stores correctly.
Choke Point B-1
Peer-to-Peer Direct Channel
Cross-validation and telemetry between peers traverses the exec zone internal network. A misconfigured firewall that blocks intra-peer traffic will cause 0% consensus on all cross-validate calls.
TEST: iptables -I INPUT -s exec-node-2 -j DROP on exec-node-1 → cross-validate → peer-alpha reports INVALID for peer-beta blocks. Remove rule → retest consensus.
Choke Point B-3
Storage Integrity Verification on Read
Every GET from FileStore re-hashes the value and compares to the stored checksum. Manual file modification after storage will trigger STORE_CORRUPT. This is the final data-integrity gate before any downstream consumer.
TEST: After telemetry stored, manually edit the .json file value field → GET /storage/peer-beta/get/<key> → confirm STORE_CORRUPT event, 404 response.

07 Acceptance Criteria

// Hard pass/fail definitions — not negotiable thresholds
IDCriterionTest ReferencePass Definition
AC-01 Hardware signing active P1.1–P1.3 HSM events contain "device":"yubihsm2". Root key not present in any env var or log output.
AC-02 Block validation succeeds P1.3 valid=true, msg=VALID returned from hardware-backed verify_epoch().
AC-03 Task execution produces signed block P2.2–P2.3 block_hash present in ACK, block exists in peer's DAG, epoch stamp on block meta.

AC-06 Cross-validation reaches 100% P3.3 consensus_pct=100.0 after block is replicated to all peers.
AC-07 Tamper detection is immediate P4.1–P4.2 validate() returns Bad signature on tampered block within one validation call.
AC-08 Network tamper drops consensus to 0% P4.3 cross_validate() returns consensus_pct=0.0 on tampered block — no peer votes valid.
AC-09 File corruption detected on read P4.5 Manually modified store file returns 404 + STORE_CORRUPT event. Not silently served.
AC-10 Choke point failure is clean Chokepoint tests Every blocked choke point produces a logged error event, not a silent timeout or crash.

If a Peer Stops Responding

Check systemctl status warlok-peer on the affected node. Check firewall rules have not changed. Check journalctl -u warlok-peer -n 50 for PEER_AUTH_FAIL or HSM errors. If HSM connector is unreachable, restart yubihsm-connector on hub-node first, then restart the peer.

Epoch Rotation Event

The HSM/Epochs page will show the epoch counter incrementing after every 10 blocks signed. This is expected. No operator action required. The new epoch key is automatically provisioned on the YubiHSM2 device. Existing blocks retain their epoch stamp and remain verifiable via verify_epoch() using their stored epoch ID.

If Cross-Validate Returns 0%

First check whether the target block has been replicated to all peers (Storage Monitor → peer record counts). If the block is in hub DAG only, call POST /peers/replicate with the block hash, then re-run cross-validate. If replication succeeds but consensus is still 0%, run POST /validate directly on the block hash — a Bad signature response indicates the block was tampered. Document the incident, preserve the event log, and escalate.

Preserving the Event Log Before Reset

Export event log before any resetShell
# Export current event log to timestamped file
DATE=$(date +%Y%m%d-%H%M%S)
curl -s https://hub-node:8000/events \
  -H "X-WARLOK-KEY: $API_KEY" \
  > /data/warlok/logs/events-$DATE.json

# Confirm count matches before reset
echo "Events saved: $(python3 -c "import json; d=json.load(open('/data/warlok/logs/events-$DATE.json')); print(d['count'])")"

# Only then reset — if required by test protocol
curl -s -X POST https://hub-node:8000/reset -H "X-WARLOK-KEY: $API_KEY"
The /reset endpoint clears all in-memory state — blocks, events, peers. It does not clear the FileStore (files on disk remain). It does not clear the YubiHSM2 — key slots and epoch keys are preserved. A reset is a process restart, not a device wipe. However, any blocks created before the reset will not be in the new hub's DAG, so cross-validation will return 0% until blocks are recreated. Export the event log first.
